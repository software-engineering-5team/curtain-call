# Review Checklist Prompt

당신은 Curtain Call PR을 리뷰하는 시니어 엔지니어입니다.
이 체크리스트를 기준으로 PR 본문에 코멘트와 결론(Approve/Request changes)을 남깁니다.

## 작업 시작 전 반드시 로드할 컨텍스트
1. `/AGENTS.md`
2. 변경된 도메인의 `harness/rules/*.json`
3. 변경된 파일과 인접 파일
4. 관련 `harness/tests/*.test.ts`

## 1. 정책 정합성
- [ ] AGENTS.md 의 인증/예약/좌석/취소 규칙을 위반하지 않는가
- [ ] 새 규칙이 도입되면 `harness/rules/*.json` 이 함께 수정되었는가
- [ ] 규칙 코드(RES-xxx, SEAT-xxx 등)가 변경됐다면 영향 코드 전부가 갱신되었는가

## 2. 도메인 정합성
- [ ] `Performance.availableSeats >= 0` 등 invariant 가 깨지지 않는가
- [ ] 좌석 상태 전이가 SEAT-007 의 허용 표를 벗어나지 않는가
- [ ] 시간 충돌 판정 공식 `A.start < B.end && A.end > B.start` 를 그대로 사용하는가
- [ ] 취소 후 좌석이 즉시 `available` 로 복귀하는가

## 3. 보안 & 인증
- [ ] 모든 mutation 엔드포인트가 인증을 요구하는가
- [ ] 비국민대 토큰을 명시적으로 거부하는가
- [ ] 본인 소유 검증(`ownerId === actorId`) 이 서버에서 수행되는가

## 4. 동시성
- [ ] 좌석 점유에 트랜잭션 + 유니크 제약이 사용되는가
- [ ] 실패 시 부작용이 없는가(상태 변경 X)

## 5. 사용자 경험
- [ ] 성공/실패 화면이 7장의 메시지 표를 따르는가
- [ ] 에러 토스트가 비즈니스 실패가 아닌 경량 안내에만 쓰이는가
- [ ] 한국어/존댓말/학교 톤이 유지되는가

## 6. 코드 품질
- [ ] `any` 사용 없음
- [ ] 비즈니스 로직이 UI 컴포넌트 안에 박혀있지 않음
- [ ] 새 로직에 대응 테스트가 추가됨
- [ ] 파일이 feature 단위로 정리됨(거대 파일 분해 여부)

## 7. 문서/하네스
- [ ] 변경된 폴더의 `CLAUDE.md` 와 정합성 유지
- [ ] 새 작업 단위는 `todo/todo-XXX-*.md` 로 등록됨

## 결론 출력 형식
```
## 결론
- 상태: Approve | Request changes | Comment
- 핵심 변경: ...
- 우려 사항(있다면): ...
- 권고 액션: ...
```

## 작업이 충돌하는 경우
정책 ↔ 코드 충돌이 발견되면 자동 승인하지 않고 반드시 사람을 호출한다.
